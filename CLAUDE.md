# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-only client for [ArtifactsMMO](https://play.artifactsmmo.com/) (Vite + TypeScript + Preact, `@preact/signals` for state). It talks **directly** to the public game API (`https://api.artifactsmmo.com`) from the browser with a Bearer token — there is **no backend**. It deploys as static files to GitHub Pages.

## Commands

```bash
npm run dev        # Vite dev server with HMR
npm run build      # production build → dist/  (BASE_PATH env var sets Vite `base`)
npm run preview    # serve the production build
npm run typecheck  # tsc --noEmit
npm run assets     # bash scripts/download-assets.sh — re-download public/assets/ icons+tiles
```

There is **no test runner** in this project. Use `npm run typecheck` as the correctness gate; CI runs it before every deploy. The app sits behind a token-entry gate, so behavior usually can't be verified headlessly — rely on typecheck + the production build (`npm run build`) succeeding.

**Dev forensics**: while the dev server runs, the app streams a trace of every API action (method, path, body, status), queue note change, head-item transition and activity-log entry to `logs/dev.log` (gitignored) — the browser batches lines to the `/__devlog` middleware in `vite.config.ts` (`src/lib/devlog.ts`; per-tab tags distinguish sessions). Read it to debug runner behavior post-hoc — it is how the potion-brew bank livelock was caught. No-op in production builds.

On Windows/Git-Bash, do **not** set `BASE_PATH=/foo/` for a local build via the Bash tool — MSYS rewrites the leading-`/` value into a Windows path. Use PowerShell (`$env:BASE_PATH='/foo/'`) if you need to test a subpath build locally. CI (Linux) is unaffected.

## The core invariant: one sync, then no polling

This is the whole design and the thing most likely to be violated by accident. Read `src/state/sync.ts` and `src/state/apply.ts` together.

- At boot (`main.tsx` → `boot()` in `sync.ts`): paint instantly from `localStorage`, load static catalogs from `public/data/*.json` into memory, then do **exactly one** authoritative read of `/my/characters`, `/my/bank`, `/my/bank/items`, `/events/active`, `/my/pending_items` (unclaimed achievement payouts → 🎁 topbar badge + claim drawer; a claim's echo updates state, the row is dropped locally), account.
- **Time-limited map events** (`/events/active` → `activeEvents` signal in `store.ts`, helpers in `src/state/events.ts`): fetched only in that sync; each event's expiration is enforced **locally** by the 4 Hz clock (no re-fetch). Event tiles overlay the static map everywhere — MapView drawing/hover/click, `liveTileAt`, `nearest()` in loopkit, and the queue's fight/gather pickers (`availableMonsters`/`availableResources`, tagged ⚡). New events appearing mid-session show up only after a manual Refresh.
- After that, **nothing polls.** Every action endpoint echoes the full authoritative `character` (and bank item moves echo `bank`), so `applyActionResult` (`src/state/apply.ts`) is the single chokepoint that folds responses into state — no GET-after-write. Re-sync only on the manual Refresh button (`reconcile()`).
- Budgets to respect: per-account **read** endpoints are the scarcest (~300/hr) — never add polling/GET loops. Actions are ~2000/hr; automation loops are paced by the cooldown each action returns, which keeps them well under.

`src/api/client.ts` is the only place that does `fetch`. It handles 429 (rate-limit) and 499 (still-on-cooldown) retries, transient network failures (transport-level backoff, 2s→30s over ~90s, then a code-0 ApiError — which the queue runner holds-and-retries on instead of pausing), and pagination, and returns the unwrapped `data`. New API calls go through `api()` / typed wrappers in `src/api/actions.ts`, which route every response through `applyActionResult`.

## State

`src/state/store.ts` holds all dynamic state as `@preact/signals` (`characters`, `bankItems`, `bankDetails`, `selectedCharacter`, `panelTarget`, a 4 Hz `now` clock for cooldown countdowns, etc.). Components read signals directly and re-render automatically. `src/state/persist.ts` mirrors the durable signals to `localStorage` (key `ammo:v1:state`) so reload paints instantly.

## The automation runner (the queue)

Exactly **one** per-character async runner executes everything: the **queue** (`src/state/queue.ts` + item vocabulary in `src/plan/queue.ts`), driven entirely by action cooldowns (`setTimeout` chains) — **it runs in the open browser tab; GitHub Pages does not run it.**

A queue is a user-editable ordered list of simple actions (move / fight ×N / gather ×N / craft ×N / withdraw / gear / accept-task / work-task / deliver / turn-in / …). `work-task` works whatever the CURRENT task turns out to be (unknown until accept-task runs): a bank-reset gear swap once per task (`geared` remembers it), then fight the task monster, or acquire the task item in the field — gather it, or gather materials + craft — handing bagfuls straight to the items master; task stock never moves through the bank (`deliver` is the from-bank-stock alternative). `task-loop` chains accept → work-task → turn-in as ONE repeating item (×N tasks, 0 = forever) — `done` counts turn-ins, `geared` re-arms the bank-reset gear swap per new task. Fight, gather and craft items with a count of `0` run **forever** (craft ∞ = withdraw-a-bagful → craft → deposit cycles until the bank can't feed the recipe). Every item is skip-if-satisfied, so reloads resume mid-item; failure **pauses** the queue with the error on the head item — EXCEPT a missing map target (monster/resource/shop/workshop/master not on the map, e.g. an expired event): that **skips** the item with a log line (`skipItem`) so queues survive events ending. Multi-character bank races never pause either: a withdraw hitting 404/478 (a sibling took the stock first) drops the code from the local `bankItems` signal and re-plans next tick (`gearSwapStep` + `runStep` in exec.ts), and 461 (concurrent bank transaction) is waited out inside `api()` like a cooldown. Key `ammo:v1:queue`.

These simple items are the **only execution vocabulary**. The old planner/campaign layer (goal compiler, task/train-craft phase machines, acquisition resolver) was removed 2026-07-06 — a planner must compile goals **down to queue items**, never run its own machinery. The hive (below) is that planner.

Stop semantics: ⏹ persists `running: false` IMMEDIATELY (a reload can never resurrect a stopped queue) while the loop drains its in-flight action — `liveLoops` covers that gap so ▶ during the drain just cancels the stop instead of double-launching. `queue.ts` self-reloads the page on HMR (a hot swap would zombie the old runLoop).

The runner pattern (copy it if a second runner ever returns): a `signal<Record<name, Job>>`, a module-level `stopFlags` Set, `step()` = run one action then `waitCooldownFull` (`src/state/loopkit.ts`), and the **reload-resume** pattern — the job signal is hydrated from its own `localStorage` key at module load, an `effect()` writes it back on change, and `resumeQueue()` (called from `boot()`/`login()` in `sync.ts` **after** the sync) re-launches. Hydrate before the persisting `effect` first runs or it overwrites the saved jobs. **Do not route runner persistence through `persist.ts`** (import cycle: `persist → queue → actions → apply → persist`) — the runner owns its key. Shared risky mechanics (bank-off, food-first healing, forecast gating, gear swaps) live once in `src/state/exec.ts`.

## The hive: the account-goal coordinator

The strategy layer above the queues — a **pure compiler + signal observer**, never a second execution engine. Domain in `src/plan/hive/` (pure functions over a `HiveCtx` snapshot: no signals, no API, no `state/` imports), runtime in `src/state/hive.ts`, UI in `src/ui/HivePanel.tsx` (🐝 topbar drawer + status strip above the roster).

- **Propose → launch → observe.** `proposeGoals` enumerates + scores account goals (gear-upgrade / farm-tasks / achievement / level-skill) as value-per-hour (`score.ts` holds every tunable); blocked goals are discounted and their prerequisites promoted as standalone goals carrying the parent's undiscounted score. A fifth kind, **craft-order** ("make me N of X" — the fleet gathers in parallel, the pinned/qualified crafter finishes behind a bank barrier), is built by the user in the drawer's order builder and never auto-proposed; it is satisfied when the fleet OWNS the ordered quantities. The user Launches one; the hive **takes over** all non-opted-out characters' queues (stop + clear, behind a confirm), then only ever writes queue items and watches the `queues` signal (one `effect` → microtask-coalesced `tick`).
- **Recompile-at-barrier is the core contract**: `compileGoal` returns waves but only `waves[0]` executes; when its assignments complete, the runtime re-calls `compileGoal` with fresh state and dispatches the NEW wave 0 — drop shortfalls become top-up waves, a consumer craft is only compiled once its producers' deposits are visible in the bank (a finite craft never pauses on missing mats), and skill training re-picks its recipe as the level rises. `HivePlan.once` (farm-tasks) means "wave done = goal done, no recompile"; an item-identical recompiled wave trips the **stall guard** (finish `incomplete`). Waves must contain **no infinite items** (they'd never complete) and ≤1 assignment per character — compile-side rules: mid-wave idle is filled at **runtime** instead (`fillWaveIdle`): a participant whose part is done (or who was never assigned) gets an infinite items/monsters task-loop as a `filler` assignment (`fillerItems` in tasks.ts picks the master) — excluded from `waveDone`, item ids in the `f` namespace, dispatched only onto an empty stopped queue (user edits win), pulled by `beginBarrier` so filler never delays the goal.
- **Dispatch is idempotent** via deterministic item ids (`hive:<startedAt>:<waveSeq>:<char>:<i>`): reloads re-attach instead of re-pushing, and a user deleting a hive item counts as done — the hive appends and watches; user edits win. Per-assignment status is **derived** from the queues signal (`assignmentStatus`: done = our ids gone, paused = error on OUR item, blocked = error on a user item ahead), never stored. A paused assignment holds its wave (others keep working) and gets Retry/Skip in the UI — no auto-retry.
- **Budget**: the hive itself performs ZERO API calls except one `loadAchievements()` per barrier for achievement goals (dynamic import of sync.ts keeps the static graph acyclic). `fleetBis` (ctx.ts) memoizes BIS searches per bank array reference with planning-grade pool caps; a cold `proposeGoals` is a 1–3 s synchronous sweep behind an explicit button.
- Follows the runner pattern: own key `ammo:v1:hive` (hydrate before the persisting effect, NOT via persist.ts), `resumeHive()` from `boot()`/`login()` **after** `resumeQueue()`, stop-sticks-first (`status:"stopped"` persists before queues unwind), HMR self-reload.
- `planAcquire`/`acquireWaves` (`plan/hive/acquire.ts`) is the rebuilt acquisition resolver — fleet-aware (gather → craft → fight → buy, candidates by skill/forecast, virtual-bank accounting, gather→smelt chains stay in one queue, deeper crafts become barrier waves). Dead ends surface as `Blocker`s with a suggested prerequisite goal (small skill gap → level-skill, tasks_coin shortage → farm-tasks).

## The gear model: bank reset, bank-only sets

Gear is **never crafted or acquired for a character** — the bank is the single source of truth. Read `src/plan/jobgear.ts` (planner + stateless differ) with `gearSwapStep` in `src/state/exec.ts`.

- **A gear-managed job starts with a full bank reset**: go to the bank, deposit EVERYTHING (whole inventory, all pocket gold, every worn slot incl. utility potion stacks), then withdraw + equip the best set **available in the bank at that moment** (`jobGear`: fights via the bank-only BIS solver, gather/craft via prospecting/wisdom/tool argmax). Implemented as the `reset` mode of `nextGearAction`; the queue's `gear` item carries `reset: true`.
- **Fight sets include utility potions**: the BIS search covers the utility slots from owned stock (restore/boost potions change the forecast honestly — the live gate sees worn stacks via `currentFighter`). The differ equips whole stacks (game cap 100) and re-fills a slot from the bank when it drains empty; non-fight desired maps leave utility slots unmanaged, and a reset merges `RESET_UTILITY_STRIP` as an **underlay** so a fight set's potion picks survive while other jobs strip them.
- **Potions get brewed when the bank runs out**: `brewablePotions` (consumables.ts) lists potions the character could produce (alchemy skill + every ingredient stocked or gatherable); jobGear runs a second BIS solve with them as `extraCandidates` and adopts the set **only on a material forecast gain** (win-flip / safe-flip / real expected-HP improvement) — the solver's fills-a-slot tiebreak alone must never send anyone on a brew expedition. `provisionPotions` (exec.ts, called from the fight item and taskGear after the swap converges) then brews a small batch (`BREW_BATCH`, ~15 — a brew means the fighter is standing around instead of fighting, so batches stay minutes-sized; the next expedition runs when the worn stack drains dry) via the shared `produceOnce` machinery, and the next gear tick equips it.
- Mid-job bank trips (inventory full) deposit loot + gold but keep worn gear (`depositAll`/`bankOff`). A **swap ceremony standing on the bank tile sweeps the whole bag first** (deposit-all-before-equipping: everything minus `keep` + unmet wants, utility wants protected up to a full stack) — in the field the stow list stays junk gear only, and loot never earns a dedicated trip.
- **Per-iteration self-heal**: fight/gather rounds re-derive the desired set from the live bank signal (memoized per bank reference — free until a bank echo changes it) and swap when something better appears.
- **Between-fight food is self-provisioned** (fight / work-task / task-loop items, `food: false` on a fight item opts out): a hurt fighter eats the best food in hand (`healOnce`), and when the hand runs dry `provisionFood` (exec.ts, plan side in `src/plan/consumables.ts`) sources more — withdraw the best stocked bank food, else **cook** the best recipe the stock can feed, else **gather** the raw materials in the field and cook them; batch sized by the fight forecast's expected HP loss (capped at ⅓ bag). Infeasible ⇒ falls back to rest — food never pauses a queue. Carried food + its ingredients ride the `keep` list so bank-offs and gear resets don't stash them; the task deliverable is never eaten.
- **Pinned artifacts** (`PINNED_ARTIFACTS` in jobgear.ts, currently `novice_guide`): once owned, every job set includes one, and a worn copy is NEVER removed — the differ drops its slot from management entirely (even under a reset/stripAll), so it survives all swaps. Spare copies still deposit. Also: artifacts must be DISTINCT (game rule, 485 "already equipped") — jobGear's argmax and the differ both dedupe artifact codes.
- `src/plan/acquire.ts` is now only the `npcForSell` catalog lookup (the full acquisition resolver went with the planner layer).

## UI layout

`app.tsx` gates on auth (`TokenGate`) then renders a full-height `.main-stage` flex row. **Character management is the main area** (`.workspace`, left, flexes): a horizontal `Roster` strip of `CharacterMini` cards on top (clicking selects + centers the map), then `CharacterPanel` — a multi-column `.ws-grid` dashboard of the selected character (control column with the Queue first and widest, then equipment/combat, then skills/inventory; columns collapse at 1560/1150px). A character is always selected (`sync.ts` defaults to the first after boot/reconcile). **The map is a right sidebar** (`.side-stack`, 420px): `MapView` (canvas, still fully interactive — pan/zoom/tile-click, `MapInspector` inside its `.map-wrap`), then `EventsPanel` (active time-limited events with countdowns; a row click centers the map via `focusTile`), then `ActivityLog`. `CatalogPanel`/`ItemsCatalogPanel` (open on clicking a workshop / NPC / **bank** tile — branches on `panelTarget.type`) render as a wide (~640px) non-modal overlay **drawer** over the sidebar (`.catalog-panel` is `position:absolute` in `.main-stage`); recipes/bank/NPC/items lists inside are responsive card grids; Escape closes the drawer.

Moving a character by hand: the 🚶 Walk button (`ActionBar`) arms `moveMode` — the next map click moves the character — and the drawer offers "🚶 Send \<selected\>" when nobody is on its tile. **Access-gated tiles** (`maps.json access.type === "conditional"`, e.g. the Tasks Trader behind the Tasks Farmer achievement): `src/state/access.ts` resolves the gate + whether it's met (from the on-demand `achievements` signal; `null` = unknown, the server stays the judge — never hard-block on it). The inspector and drawer show a 🔒 requirement line (the drawer lazily fetches achievements to color it), and `actions.move` rethrows the server's bare "conditions not met" with the requirement spelled out.

## Static catalog

`public/data/*.json` is a committed snapshot of the game's static data (items, maps, monsters, resources, npcs, …). `src/catalog/` loads it once into typed `Map`s and exposes sync lookups (`item()`, `monster()`, `mapAt()`, `tileAt()`, `itemName()`, …) that safely return `undefined` before load. `public/assets/` (map tiles + icons, committed) is served locally; most `<img>`s fall back to the CDN via `assetFallback`, **but map tiles in `MapView` have no fallback** — they must ship in the build or the map renders blank.

## Deploy

`.github/workflows/deploy.yml` builds and deploys to GitHub Pages on push to `main`. `vite.config.ts` sets `base: process.env.BASE_PATH || "/"`; the workflow passes `BASE_PATH=/<repo-name>/` so the project-subpath build resolves (all asset URLs go through `import.meta.env.BASE_URL`). The deployed build ships **no token** — `.env` is gitignored and never set in CI; users paste their token into the in-browser gate, stored only in their own `localStorage`.
